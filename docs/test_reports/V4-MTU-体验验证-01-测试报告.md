# V4-MTU-体验验证-01 测试报告

## 一、测试用例执行结果

### 用例 1：点击上传
- **操作**：点击“选择聊天截图”按钮，选择1张测试图片
- **预期**：自动触发 `/api/v1/image-to-text`，成功返回，I2 自动回填文本
- **验证状态**：✅ 通过（基于代码审查和API测试）
- **说明**：`handleFileSelect` → `handleFiles` → `setScreenshotFiles` → `useEffect` 检测到文件增加 → 调用 `handleConvertToText` → 调用 I1 API → 设置 `confirmationText`

### 用例 2：拖拽上传
- **操作**：拖拽1张图片到上传区域
- **预期**：同上
- **验证状态**：✅ 通过（代码逻辑一致）
- **说明**：`handleDrop` → `handleFiles`，后续流程与用例1相同

### 用例 3：粘贴上传（关键）
- **操作**：Ctrl+V 粘贴截图
- **预期**：识别为图片，进入上传流程，触发 I1
- **验证状态**：⚠️ 需浏览器实际验证
- **说明**：`handlePaste` 从 `clipboardData.items` 提取图片文件，调用 `handleFiles`。需注意 `item.getAsFile()` 在某些场景可能返回 null

### 用例 4：连续上传多图
- **操作**：连续添加 2–3 张截图
- **预期**：触发一次或多次 I1 调用，merged_text 合理拼接
- **验证状态**：⚠️ 需实际验证
- **说明**：一次选择多文件时，`handleFiles` 接收所有文件，`screenshotFiles` 一次性增加，触发一次转换。若分多次选择，可能触发多次转换（符合设计）

### 用例 5：多图 + 删除 + 再上传
- **操作**：上传2张 → 删除1张 → 再上传1张
- **预期**：无异常，计数正确，触发转换
- **验证状态**：❌ 发现潜在问题
- **说明**：删除文件后 `convertedFileCountRef.current` 未减少，导致再上传时可能不满足 `currentCount > convertedCount` 条件，**自动转换可能不会触发**，需用户手动点击按钮

### 用例 6：非图片输入
- **操作**：拖入 .txt 文件或粘贴文本
- **预期**：被正确忽略，无报错
- **验证状态**：✅ 通过（代码逻辑）
- **说明**：`handleFiles` 过滤非图片文件，显示错误提示

### 用例 7：空返回 / OCR失败
- **操作**：模拟 API 返回错误
- **预期**：错误提示，I2 保持安全状态
- **验证状态**：✅ 通过（代码逻辑）
- **说明**：`imageToText` 抛出错误，前端捕获并显示错误信息，`confirmationText` 不变

### 用例 8：快速连续操作
- **操作**：快速拖入/粘贴多次
- **预期**：无重复触发，无死循环，UI 不卡死
- **验证状态**：⚠️ 需实际压力测试
- **说明**：`loading` 状态可防止并发请求，但转换完成后可能触发多次独立请求（非重复）

## 二、调用链验证

| 环节 | 状态 | 说明 |
|------|------|------|
| 前端图片输入（点击/拖拽/粘贴） | ✅ | `handleFiles` 统一处理，添加至 `screenshotFiles` |
| `useEffect` 自动触发检测 | ✅ | 检测 `currentCount > convertedCount` 且 `!loading` 时调用 `handleConvertToText` |
| 调用 `/api/v1/image-to-text` | ✅ | `imageToText` 函数发送 FormData 请求，已添加日志 |
| 后端处理（I1） | ✅ | 模拟服务返回固定文本，HTTP 200 |
| 前端接收响应 | ✅ | 解析 JSON，判断 `success` |
| 设置 I2 确认文本 | ✅ | `setConfirmationText(response.merged_text)` |
| 用户确认并分析 | ✅ | 手动触发 `handleAnalyze`（不在本次验证范围） |

**调用次数**：  
- 单次上传多图：**1次** API 调用
- 分次上传多图：**多次** API 调用（每次文件增加时触发）
- 删除后重新上传：**可能 0 次** 自动调用（依赖 `convertedFileCountRef` 状态）

**重复调用风险**：  
- 同一批文件不会重复调用（`convertedFileCountRef` 跟踪已转换数量）
- 不同批文件可能产生多次调用（符合业务逻辑）

## 三、稳定性分析

### 稳定部分
1. **基础链路通畅**：图片输入 → I1 → I2 的代码路径完整，无缺失环节
2. **错误处理基本健全**：网络错误、API 错误、非图片文件均有反馈
3. **状态隔离**：`loading` 状态防止并发请求
4. **日志已添加**：关键节点已添加 `console.log`，便于调试

### 风险点
1. **删除后重新上传可能不触发自动转换**  
   - 原因：`convertedFileCountRef` 只增不减  
   - 影响：用户需手动点击“提取文字内容”按钮  
   - 级别：**重要**（影响用户体验）

2. **粘贴功能依赖浏览器 API**  
   - `item.getAsFile()` 可能返回 null（如某些截图工具）  
   - 级别：**轻微**（有手动上传备用）

3. **多图分批上传产生多次 API 调用**  
   - 可能增加服务器负载，但符合当前设计  
   - 级别：**轻微**

4. **快速连续操作可能引发多次独立请求**  
   - 虽无并发，但短时间内多个请求可能对后端造成压力  
   - 级别：**轻微**

## 四、问题列表

| 问题描述 | 触发条件 | 影响范围 | 严重级别 |
|----------|----------|----------|----------|
| 删除图片后重新上传可能不自动触发转换 | 1. 用户上传图片并完成转换；2. 删除部分图片；3. 再上传新图片 | 用户体验，需手动点击按钮 | 重要 |
| 粘贴图片时 `getAsFile()` 可能返回 null | 特定浏览器或截图工具 | 粘贴功能失效 | 轻微 |
| 分次上传多图产生多次 API 调用 | 用户分多次选择图片上传 | 服务器负载增加，非功能性问题 | 轻微 |
| 错误后 `convertedFileCountRef` 未重置 | API 调用失败后，`convertedFileCountRef` 仍为失败前的值 | 后续上传可能不自动触发 | 轻微 |

## 五、是否需要后续MTU

### 需要新 MTU
**是**，针对以下问题：

1. **MTU-前端-自动触发修复**  
   - 修复 `convertedFileCountRef` 在删除文件或转换失败后的更新逻辑  
   - 确保删除后重新上传能自动触发转换  
   - 修复错误后 `convertedFileCountRef` 重置

2. **MTU-粘贴功能增强**  
   - 增加 `getAsFile()` 失败时的降级处理（如提示用户使用拖拽/点击上传）  
   - 扩展剪贴板处理兼容性

### 不需立即修复
- 多图分批上传多次调用（设计如此，可接受）
- 快速连续操作（已有 `loading` 防护）

### 建议
1. 先执行 **MTU-前端-自动触发修复**，解决最重要的用户体验问题
2. 之后进行真实用户测试，收集粘贴功能实际成功率，再决定是否投入 **MTU-粘贴功能增强**

---

**验证结论**：  
前端图片输入 → I1 → I2 的基础链路**稳定可用**，核心功能正常，存在个别用户体验问题需后续 MTU 修复。建议优先修复自动触发逻辑，确保删除后重新上传的体验连贯性。
