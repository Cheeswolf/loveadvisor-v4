《LoveAdvisor V4 架构冻结文档（正式版）》
一、冻结目的

V4 不是继续“边写边看”的阶段，而是从工程系统进入产品系统的阶段。
因此，V4 在正式开发前，必须先冻结两类架构：

1. 系统架构
2. 项目文件架构

冻结后，任何开发任务都只能在该架构边界内推进，CloudCode 不得自由扩散。

二、V4 阶段定义

V4 的本质不是增强分析能力，而是把已可用的分析引擎包装成可供真实用户试用的产品闭环。这一点在现有 V4 方案中已经明确：V4 是从“分析引擎系统”升级为“可试点产品系统”的关键版本，目标是构建可供真实用户试用的完整产品闭环。

因此，V4 的主目标固定为：

真实用户入口 + 输入闭环 + 展示闭环 + 数据闭环

而不是：

继续重做 R1 / analyze / pipeline
三、V4 系统总体架构（冻结）

V4 系统架构正式冻结为 5 层：

1. 用户入口层
2. 输入处理层
3. 核心分析引擎层
4. 产品数据层
5. 商业化预留层

这 5 层来自现有 V4 方案，现正式作为唯一有效版本冻结。

四、五层架构的正式定义
4.1 用户入口层

职责：

面向真实用户提供产品入口
承担文本输入、截图上传、I2 文本确认、结果展示
承担后续产品交互壳

正式实现：

轻前端（Next.js / React）

这里必须明确，V4 架构里新增的“轻前端”已经被正式写入用户入口层，不是临时补丁。方案原文已明确：用户入口层包含轻前端（Next.js）、截图上传 / 文本输入、I2 文本确认、结果展示。

4.2 输入处理层

职责：

管理文本输入路径
管理截图输入路径
承接 I1 图转文与 I2 用户确认
产出 confirmed_chat_text

正式输入链路冻结为：

截图 → I1 → I2 → confirmed_chat_text
文本 → 直接进入 confirmed_chat_text 路径
4.3 核心分析引擎层

职责：

只处理已确认文本
完成关系信号提取、规则推断、建议生成与输出组装

正式冻结链路：

PREPROCESS → S2 → S3 → R1 → S5 → GUARDRAIL → OUTPUT

该主链路来自现有 V4 方案，并继续作为核心资产冻结。

4.4 产品数据层

职责：

保存用户分析记录
保存使用配额与反馈
支撑后续 MVP+ 阶段的数据闭环

现阶段只允许新增，不允许反向侵入分析引擎。

4.5 商业化预留层

职责：

为登录、套餐、支付保留接口与结构位

现阶段不实现，只允许占位，不允许投入主要开发资源。现有 V4 方案也明确把商业化层定义为“预留，不实现”。

五、双前端定位（正式冻结）

V4 开始，前端正式采用双前端定位：

A. 正式用户端前端
B. 内部调试前端
5.1 正式用户端前端

职责：

面向真实用户
承担产品入口与交互
承担输入闭环与展示闭环

正式技术栈：

Next.js / React
5.2 内部调试前端

职责：

本地联调
协议验证
I1 / I2 / analyze 路径调试
快速验证结果展示

正式技术栈：

Streamlit
5.3 冻结结论
Next.js / React = 正式用户端
Streamlit = 内部调试端

这条边界从现在开始固定，不再摇摆。
也就是说，后续不再把 Streamlit 当作正式用户前端打磨；它保留，但仅用于内部调试与联调。

六、核心引擎冻结边界（强制）

现有 V4 方案已经给出了硬冻结区，现直接升格为强制规则：

八、根目录硬规则（强制）

从现在开始，根目录只允许出现以下类型：

项目入口文件：run.py
依赖文件：requirements.txt
环境文件：.env
说明文件：README.md
一级工程目录：app/、web/、frontend/、configs/、test_system/、scripts/、data/、docs/

以下内容禁止直接落在根目录：

test_*.py
debug_*.py
temp_*.py
调试输出 *.json
测试输出 *.txt
临时日志与中间结果文件

这条规则是本次清场的直接结论：清场任务的核心就是把这些文件从根目录彻底移除。

九、测试与调试文件落点规则（强制）
9.1 正式测试文件

凡是能重复执行、用于标准验证的测试脚本，必须放在：

test_system/scripts/
9.2 临时调试测试脚本

凡是临时性的、问题定位类、针对单一案例的脚本，必须放在：

test_system/debug_tools/
9.3 测试输出

凡是批测结果、统计结果、测试过程输出，必须放在：

test_system/output/
9.4 历史归档

scripts/debug_archive/ 只用于保存历史清场归档，不再作为新的开发落点。
这点必须单独强调，否则它会重新演变成第二个垃圾堆。

十、CloudCode 落文件纪律（正式冻结）

从本文件生效开始，CloudCode 必须遵守以下规则：

10.1 不允许自由创建根目录文件

未在本文件中定义落点的文件，不允许直接创建在项目根目录。

10.2 先定目录，再写代码

任何新增模块，必须先明确归属目录，再允许施工。

10.3 新问题 = 新任务

不得在当前任务中顺手新增脚本、顺手加 debug 文件、顺手产生日志输出到任意目录。

10.4 测试只能进 test_system

任何测试、回归、样本验证脚本，都必须落入 test_system/ 体系。

10.5 调试不能污染正式模块

调试用途文件不得落入 app/、web/、frontend/ 根层，除非是明确受控的日志或调试开关。
