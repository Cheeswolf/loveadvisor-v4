# 《LoveAdvisor V3.6 I1 图转文节点最小复杂度技术方案》

---

## 一、方案目标

本方案只解决一个问题：

> **让“聊天截图输入”从占位演示，升级为真实可用的图转文输入能力。**

且必须满足以下前提：

1. **不重构现有 V3 分析引擎**
2. **不修改 rules / pipeline / analyze 主职责**
3. **所有图转文结果必须先经过 I2 用户确认**
4. **Claude Code 施工必须保持 MTU 纪律**   

---

## 二、核心裁决

### 2.1 本阶段最优路线

V3.6 第一版 I1 的最优路线定义为：

```text
云端视觉模型读图
→ 返回文本初稿
→ I2 用户确认
→ confirmed_chat_text
→ 进入现有 analyze
```

### 2.2 为什么这是“最小复杂度最优解”

因为它同时满足四个条件：

| 维度       | 结论 |
| -------- | -- |
| 架构稳定性    | 高  |
| 接入复杂度    | 低  |
| 对聊天截图适配性 | 高  |
| 后续可替换性   | 高  |

它避免了两种错误：

第一，**把 OCR 直接塞进 analyze**。
这会打穿你已经冻结的输入层与分析层边界。

第二，**第一版就做复杂 OCR 后处理系统**。
这会把“图转文节点”做成一个新的复杂引擎，直接违背“最小复杂度”原则。

---

## 三、最小化系统设计

---

## 3.1 系统定位

### I1 的一句话定义

```text
I1 = 图片 → 文本草稿生成器
```

它不是：

* 关系分析器
* 最终文本定稿器
* 说话人最终判定器
* 跳过确认层的快捷通道

这个定位与你已经冻结的 V3.6 架构完全一致。

---

## 3.2 最小链路

```text
前端上传截图
→ POST /api/v1/image-to-text
→ 后端 I1 调用视觉模型
→ 返回 raw_text + merged_text
→ 前端展示到 I2 编辑区
→ 用户确认
→ confirmed_chat_text
→ POST /api/v1/analyze
```

---

## 3.3 最小边界

### 本方案明确不做

* 不修改 `/api/v1/analyze`
* 不修改 `rules.py`
* 不修改 `pipeline_orchestrator.py`
* 不把图片输入直接送进分析引擎
* 不在 I1 中做关系判断
* 不在 I1 中做复杂说话人纠错系统
* 不做本地 OCR 自研
* 不做多 provider 并发调度

这与 V3.6 的冻结边界一致。 

---

# 四、接口方案

---

## 4.1 新接口定义

建议新增独立接口：

```text
POST /api/v1/image-to-text
```

### 为什么必须独立

因为：

| 接口                      | 职责      |
| ----------------------- | ------- |
| `/api/v1/image-to-text` | 图片转文字   |
| `/api/v1/analyze`       | 文本转分析结果 |

如果复用 analyze，会导致：

* 输入协议混乱
* 错误定位困难
* 前端流程耦合
* 后续 provider 替换困难

这也是你在 V3.6 文档里已经明确裁决过的。

---

## 4.2 输入协议

第一版建议支持：

```json
{
  "images": ["<base64_or_uploaded_file_ref>", "<base64_or_uploaded_file_ref>"]
}
```

### 最小原则

第一版只支持：

* jpg / jpeg / png
* 单张或多张
* 按前端上传顺序处理

### 不做的事

* 不做 PDF
* 不做视频
* 不做拖拽排序增强
* 不做复杂元数据

---

## 4.3 输出协议

严格沿用你已经冻结的 I1 协议，只做必要细化：

```json
{
  "success": true,
  "raw_text": "识别得到的原始文本",
  "merged_text": "整理后的聊天文本初稿",
  "provider": "vision_model_v1",
  "source_type": "image",
  "image_count": 2,
  "need_user_confirm": true,
  "error_message": ""
}
```

### 字段原则

`raw_text`：保留原始识别结果，便于 debug
`merged_text`：给 I2 直接展示和编辑
`need_user_confirm`：固定 `true`
`provider`：保留，为后续替换 provider 留口子

这个双字段设计非常关键，因为它能把“识别问题”和“草稿整理问题”分开。

# 五、后端最小实现方案

---

## 5.1 新增模块建议

只新增一个非常薄的 I1 服务层，不碰分析主链路。

建议新增：

```text
app/api/image_to_text.py
app/services/image_to_text_service.py
app/schemas/image_to_text_models.py
```

### 职责拆分

#### `app/api/image_to_text.py`

只做：

* 接收上传图片
* 调用 service
* 返回协议化结果

#### `app/services/image_to_text_service.py`

只做：

* 调用视觉模型
* 合并多图识别文本
* 构造 `raw_text` 与 `merged_text`

#### `app/schemas/image_to_text_models.py`

只做：

* 请求模型
* 响应模型

---

## 5.2 不新增的模块

第一版不建议新增：

* 独立 OCR provider 工厂
* 复杂缓存层
* 图像预处理流水线
* 识别质量评分系统
* 队列任务系统

原因很简单：
这些都不是“真实可用第一版”的必要条件。

---

## 5.3 最小服务逻辑

`image_to_text_service.py` 内部逻辑只保留五步：

```text
1. 接收图片列表
2. 逐张调用视觉模型识别
3. 拼接 raw_text
4. 生成 merged_text
5. 返回统一结果
```

### merged_text 的最低整理规则

第一版不要做智能重建，只做轻整理：

* 保留识别文字
* 去掉明显空白噪声
* 多图按上传顺序拼接
* 尽量保留换行
* 不强行补 A/B 说话人

### 为什么不补 A/B

因为这一步属于**确认层之后**的事情，不应在 I1 里过度推断。
I1 的职责是“草稿生成”，不是“替用户定稿”。

---

# 六、前端最小实现方案

---

## 6.1 前端职责

前端只做三件事：

1. 上传截图
2. 调用 `/api/v1/image-to-text`
3. 把 `merged_text` 填进 I2 文本编辑区

---

## 6.2 前端不做什么

第一版前端不做：

* 客户端 OCR
* 图片裁剪增强
* 智能说话人修正
* 多页复杂管理
* 图像识别结果高亮对照

---

## 6.3 前端最小交互流程

```text
用户选择截图
→ 点击“转成聊天文本”
→ 等待 I1 返回结果
→ 把 merged_text 填入编辑框
→ 用户修正
→ 点击“开始分析”
```

### 失败兜底

如果 I1 返回失败：

* 显示错误提示
* 保留手动粘贴文本入口
* 不阻断整页使用

这是最重要的兜底策略之一。
也就是说，**截图输入失败时，文本输入路径仍然可用**。

---

# 七、provider 选择策略

---

## 7.1 第一版只允许一个 provider

为了避免复杂度爆炸，第一版策略必须是：

```text
单 provider
单接口
单流程
```

### 推荐命名

```text
provider = vision_model_v1
```

不要第一版就做：

* OpenAI / Google / Azure 同时接
* provider 自动切换
* 多模型降级链
* A/B 选择器

这些都会让 Claude Code 施工边界迅速失控。

---

## 7.2 为什么先不抽象 provider_factory

因为现在的目标是：

> **先把 I1 节点落地为真实可用能力**

不是做“图转文平台框架”。

等 I1 跑通之后，再考虑第二阶段做：

* provider 抽象
* OCR A/B 测试
* 降本路线
* 本地 OCR fallback

第一版先做抽象，收益小，复杂度高。

---

# 八、错误处理最小方案

---

## 8.1 必须处理的错误

第一版只处理以下四类：

### E1：图片为空

返回：

* `success = false`
* `error_message = "未收到图片"`

### E2：识别失败

返回：

* `success = false`
* `error_message = "图转文失败"`

### E3：识别结果为空

返回：

* `success = false`
* `error_message = "未识别到有效文字"`

### E4：部分图片失败

第一版最简单处理：

* 只要有一张失败，整体失败
* 不做部分成功合并

### 为什么这么保守

因为“部分成功 + 部分失败”的处理会引入：

* 多图状态同步
* 缺页风险
* 用户误判输入完整性

第一版宁可保守失败，也不要做脏成功。

---

## 8.2 不处理的错误

第一版暂不处理：

* 复杂截图顺序纠正
* 文本区域自动裁切
* 图片方向矫正
* 说话人角色自动恢复
* 表情/图片消息特殊结构识别

# 十、与 I2 的衔接规则

---

## 10.1 强制规则

```text
I1 输出不能直接进入 analyze
必须先进 I2
```

这是 V3.6 最核心的纪律之一。 

---

## 10.2 I2 的唯一职责

I2 继续保持：

* 展示 `merged_text`
* 允许用户编辑
* 产出 `confirmed_chat_text`

### 这一步为什么不能省

因为聊天截图天然有误识别风险。
如果跳过 I2，错误输入会直接污染关系分析结果，破坏你前面已经收口好的 V3 主链路。

# 十三、最终裁决

这套“最小复杂度技术方案”的本质是：

> **把 I1 当成一个薄节点，而不是一个新系统。**

所以它的正确做法不是：

```text
做一个很强的 OCR 子系统
```

而是：

```text
做一个最薄的图片→文本草稿入口
并强制依赖 I2 完成最终定稿
```

这和你 V3.6 的架构判断是完全一致的：
I1 是新增输入节点，I2 不能被替代，analyze 必须冻结。
