# 一、补丁说明（必须阅读）

本补丁为 LoveAdvisor V3 阶段新增的**开发施工层规范**。

---

## 1.1 引入原因

在 V3 开发后期，系统出现以下问题：

* Claude Code 单次任务过大（多目标混合）
* 输入上下文冗余（token 溢出）
* 阶段边界被打穿（跨模块修改）
* debug 与日志无分层（干扰执行）
* 验收机制弱化（连续修改导致不可控）

---

## 1.2 核心结论

```text
当前问题不属于产品架构问题，而属于施工层问题
```

即：

| 层级                 | 状态   |
| ------------------ | ---- |
| 产品层（LoveAdvisor系统） | ✅ 成立 |
| 施工层（Claude Code使用） | ❌ 失控 |

---

## 1.3 本补丁目标

```text
将 Claude Code 从“自由生成器”
→ 转变为“强约束执行器”
```

---

# 二、系统角色定义（必须遵守）

---

## 🟢 项目 Owner（你）

负责：

* 定义目标
* 执行任务
* 提供运行结果
* 最终决策

---

## 🔵 ChatGPT（架构控制层）

负责：

* 拆解任务（MTU）
* 输出标准任务单
* 控制修改边界
* 验收结果
* 维护系统一致性

---

## 🟣 Claude Code（执行层）

定义：

```text
Claude Code = 执行器（Executor）
不是设计者（Designer）
```

---

## ❗Claude Code 禁止行为

* ❌ 修改规则层（rules）
* ❌ 修改系统架构
* ❌ 自行扩展功能
* ❌ 顺手修复未指派问题
* ❌ 跨模块修改
* ❌ 推测需求执行

---

# 三、三大核心控制体系

---

# 3.1 行为控制（防乱改）

---

## 核心原则

```text
一次任务只允许做一件事
一次任务只允许修改一个模块
```

---

## 修改范围规则

Claude Code 只能修改：

```text
任务明确指定的文件
```

---

## ❌ 禁止：

* 修改未提及文件
* 自动重构
* 优化未要求部分
* 添加额外逻辑

---

## 输出规则（强制）

必须返回：

1. 修改文件列表
2. 每个文件修改说明
3. 未完成项

---

---

# 3.2 输入控制（防 token 爆炸）

---

## 输入结构（必须使用）

```text
【Claude Code执行任务】

任务名称：
目标：

上下文（最小化）：

执行要求：
1.
2.

涉及文件：
- xxx.py（必要片段）

限制：
- 不允许 xxx

输出要求：
1.
2.
3.
```

---

## 上下文最小化原则

---

### 允许输入：

* 当前任务目标
* 必要代码片段（≤30行）
* 核心报错（≤10行）

---

### ❌ 禁止输入：

* 全量日志
* 历史对话
* 全项目代码
* 多阶段背景说明

---

## 输入预算（必须遵守）

```text
文本 ≤ 800字
代码 ≤ 30行
日志 ≤ 10行
```

---

## 裁剪优先级

```text
保留：
1. 错误信息
2. 当前目标
3. 关键代码

删除：
4. 历史背景
5. 无关日志
6. 解释性内容
```

---

---

# 3.3 任务控制（防设计失控）

---

# 🔥 核心概念：MTU（最小任务单元）

---

## 定义

```text
MTU = Minimum Task Unit
```

---

## 必须满足：

```text
1. 只解决一个问题
2. 只作用一个模块
3. 只修改一类逻辑
4. 可独立验证
```

---

## ❌ 错误任务

```text
修复 parser + 优化结构 + 加日志
```

---

## ✅ 正确任务

```text
修复 s2_parser JSON解析错误
```

---

# 四、任务拆解标准（四步法）

---

## Step 1：唯一动词

任务只能包含一个动词：

* 修复 / 新增 / 重构 / 接入

---

## Step 2：唯一模块

```text
一个任务只能作用一个模块
```

---

## Step 3：明确范围

必须指定：

* 文件
* 函数
* 问题类型

---

## Step 4：验收标准

必须定义：

```text
什么情况算完成
```

---

# 五、阶段推进机制

---

## 正确流程

```text
Phase N
→ Task 1（MTU）
→ 验收
→ Task 2（MTU）
→ 验收
→ Phase完成
```

---

## ❌ 禁止

```text
一次性做多个改动再一起调试
```

---

# 六、阶段冻结机制（关键）

---

## 每个阶段必须：

```text
1. 所有任务完成
2. 全部验收通过
3. 输出结果
4. 禁止回头修改
```

---

## ❗原则

```text
新问题 = 新任务
```

---

# 七、Debug 与日志分层

---

## 三层结构

| 层级  | 用途      | 是否给 Claude Code |
| --- | ------- | --------------- |
| 核心层 | 任务 + 错误 | ✅               |
| 中间层 | 调试输出    | ❌               |
| 全量层 | 日志归档    | ❌               |

---

## 核心原则

```text
Claude Code 只看核心层
```

---

# 八、任务失败机制

---

## 当任务不明确：

Claude Code 必须输出：

```text

```

---

## ❗禁止：

* 猜测执行
* 自动补全需求
* 扩展任务范围
