# 📘《LoveAdvisor × Claude Code 协作规范 v1》

> 适用阶段：V3（本地可控化与产品化重构阶段）
> 目标：让 Claude Code 成为“工程执行层”，而不是“架构决策者”

---

# 一、项目定位（必须理解）

## 1.1 项目本质

LoveAdvisor 是一个：

> **规则主导 + LLM辅助的情感分析与恋爱决策系统**

核心原则：

* ❗规则决定结论（R1）
* ❗模型只做提取与表达（S2/S3/S5）
* ❗评测系统决定优化方向（test_system）

---

## 1.2 当前阶段（V3）

当前目标：

| 编号 | 目标              |
| -- | --------------- |
| G1 | 主链路去 Coze 化     |
| G2 | 模块本地化（S2/S3/S5） |
| G3 | 评测闭环内生化         |
| G4 | 产品化底座           |

---

## 1.3 Claude Code 的角色

Claude Code **不是开发者本人**，而是：

> ✅ **工程执行代理（Engineering Executor）**

允许做：

* 写代码
* 改代码
* 跑测试
* 修 bug
* 重构模块

禁止做：

* ❌ 改业务逻辑核心原则
* ❌ 改输出协议
* ❌ 擅自优化“恋爱策略逻辑”
* ❌ 引入未授权架构

---

# 二、系统核心架构（必须遵守）

## 2.1 主链路（不可改变）

```
input
↓
preprocess_service
↓
S2（基础信号）
↓
S3（高级信号）
↓
R1（规则引擎）
↓
S5（策略生成）
↓
guardrail（边界控制）
↓
output_builder
↓
response
```

---

## 2.2 职责边界（极其重要）

| 层级        | 是否允许 LLM | 说明           |
| --------- | -------- | ------------ |
| S2        | ✅        | 提取基础信号       |
| S3        | ✅        | 提取高级信号       |
| R1        | ❌        | **必须本地规则判定** |
| S5        | ✅        | 生成分析与建议      |
| guardrail | ❌        | 规则约束         |

---

## 2.3 核心原则

### 原则1：R1绝对主导

* relationship_stage 必须由规则决定
* interest_level 必须由规则决定

👉 Claude Code **不得修改判定逻辑方向**

---

### 原则2：S5必须服从R1

例如：

| R1结果 | S5限制   |
| ---- | ------ |
| 初识期  | 禁止表白   |
| 冷淡期  | 禁止推进关系 |
| 无法判断 | 必须保守   |

---

### 原则3：禁止“模型代替规则”

严禁出现：

```python
# ❌ 错误行为
if llm_output["stage"] == "暧昧期":
```

---

# 三、不可修改项（红线）

Claude Code **永远不能做的事情**：

### 3.1 输出协议

```json
  "relationship_stage": "...",
  "interest_level": "...",
  "key_signals": [],
  "signal_summary": [],
  "psychological_analysis": "",
  "risk_points": [],
  "suggestions": [],
  "next_step": ""
}
```

---

### 3.2 测试数据

禁止修改：

```
test_system/data/test_cases.json
```

---

### 3.3 评测指标逻辑

不得修改：

* stage_accuracy
* r1_strict_accuracy
* next_step_conservative_rate

---

### 3.4 R1判断边界（未经授权）

不得擅自改变：

* 阶段划分逻辑
* 兴趣强度判断标准
* 冷淡期定义

---

# 五、标准工作流程（必须遵守）

## 5.1 开发流程

Claude Code 必须按以下流程执行：

### Step 1：理解任务

* 阅读相关模块
* 输出理解摘要

### Step 2：提出修改计划

* 列出将修改的文件
* 说明修改目的

### Step 3：执行修改

* 逐文件修改

### Step 4：运行测试

```bash
python test_system/scripts/batch_test.py
```

### Step 5：输出报告

必须包含：

* 修改内容
* 测试结果变化
* 新问题

---

## 5.2 禁止行为

* ❌ 直接大规模改代码不说明

* ❌ 不跑测试直接结束
