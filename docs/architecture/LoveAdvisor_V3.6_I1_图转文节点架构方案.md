**文档版本**: 1.0  
**冻结日期**: 2026-04-14  
**适用阶段**: V3.6 Phase 1 (I1 图转文节点架构冻结)  
**后续依赖**: I2 确认层开发、分析引擎衔接、前端输入层改造  

---

## 1. V3.6 单独立项背景

### 1.1 从 V3.5 到 V3.6 的演进

V3.5 Phase 1 完成了输入层方案的概念设计，明确了双模输入体系（文本直接输入 + 截图转写输入）的整体框架。V3.6 在此基础上，聚焦于 **I1 图转文节点** 的架构冻结与工程化定义，确保该节点作为独立、稳定的输入通道，而非临时功能模块。

### 1.2 单独立项的必要性

1. **输入源扩展战略需求**：图片转文字是情感分析领域的核心输入场景，需要独立、鲁棒的解决方案
2. **技术栈独立性**：OCR/图像识别技术栈与传统 NLP 分析引擎技术栈分离，降低耦合
3. **迭代演进路径**：I1 节点需要独立的技术选型、性能优化、准确率提升路线图
4. **产品化门槛**：作为面向最终用户的核心功能，必须达到生产可用标准

### 1.3 本阶段目标

V3.6 Phase 1 的核心目标是 **I1 图转文节点架构冻结**，明确：
- I1 节点的工程边界与输出协议
- 与 I2 确认层的衔接接口
- 与现有分析引擎的融合方式
- 后续分阶段实施路线图

> **架构原则**：本阶段仅完成架构定义与接口规范，不绑定具体 OCR 技术实现，为后续技术选型保留灵活性。

## 2. I1 图转文节点：核心输入节点定位

### 2.1 节点性质：永久性输入通道

I1（Image-to-Text）节点是 LoveAdvisor 系统的 **第一层输入处理器**，负责将图像类输入转换为标准化文本输入。其定位为：

- **永久性架构组件**：非临时性功能，作为系统核心输入通道之一长期存在
- **独立技术域**：专注于图像识别与文字提取，技术栈独立于后续分析引擎
- **可替换实现**：接口抽象允许更换底层 OCR 引擎（云端服务/本地模型/混合方案）
- **质量演进基线**：具备独立的准确率指标、性能监控、优化路线图

### 2.2 与临时功能的区别

| 维度 | 临时功能 | I1 节点 |
|------|----------|---------|
| 架构位置 | 附加模块 | 核心输入层 |
| 接口稳定性 | 可能频繁变更 | 冻结的标准化接口 |
| 技术债务 | 允许快速实现 | 要求工程化实现 |
| 演进路径 | 一次性实现 | 持续迭代优化 |
| 质量要求 | 可用即可 | 生产级 SLA |

### 2.3 输入场景覆盖

I1 节点设计覆盖以下输入场景：
1. **单张聊天截图**：最常见的微信/QQ聊天记录截图
2. **多张连续截图**：同一聊天会话的多屏截图
3. **混合内容截图**：包含文字、表情、图片的混合内容
4. **轻度变形图像**：旋转、透视变形、低对比度等常见质量退化

> **场景排除**：本阶段不覆盖手写体、复杂版面（如报纸）、视频帧序列等高级场景。

## 3. I1 / I2 / Analyze 三层职责边界

### 3.1 三层架构总览

```
[图像输入] → I1(图转文) → [中间文本] → I2(确认清洗) → [标准化文本] → Analyze(情感分析)
```

### 3.2 I1 图转文层核心职责

1. **图像预处理**
   - 格式验证（支持 PNG、JPG、JPEG）
   - 尺寸标准化（长边 ≤ 2000px，保持比例）
   - 图像质量增强（去噪、对比度调整、二值化）

2. **光学字符识别**
   - 文字区域检测与定位
   - 多语言识别（中文为主，英文为辅）
   - 说话人标记识别（`A:`、`B:`、`【】`等常见格式）
   - 置信度分数计算（字符级/区域级）

3. **文本结构化输出**
   - 保持原始版面行序
   - 标注说话人标记位置
   - 输出置信度元数据
   - 标识识别语言

4. **错误处理与降级**
   - 识别失败时明确错误类型（图像质量、文字密度、语言不支持等）
   - 提供降级建议（如"建议截图文字更清晰区域"）

**边界约束**：
- 不进行语义理解
- 不进行情感分析
- 不进行内容真实性判断
- 不关联多图上下文

### 3.3 I2 输入确认与清洗层核心职责

1. **文本标准化**
   - 说话人标记统一化（`A:` → `A：`，`【A】` → `A：`）
   - 标点符号规范化（英文标点 → 中文标点）
   - 去除无关噪声（时间戳、系统通知、非对话内容）

2. **格式验证**
   - 最小长度验证（≥ 10字符）
   - 说话人数量验证（≥ 2个）
   - 对话内容有效性（非纯表情/符号）

3. **用户修正记录**
   - 记录原始 OCR 文本
   - 记录用户编辑差异
   - 提供优化反馈数据

**边界约束**：
- 不进行语义分析
- 不修改对话内容实质
- 不进行个性化过滤

### 3.4 Analyze 情感分析层职责

1. **信号提取**（S2/S3）
   - 情感倾向识别
   - 关系阶段判断
   - 兴趣等级评估

2. **策略生成**（S5）
   - 行动建议生成
   - 对话策略推荐
   - 风险提示

3. **输出组装**（O1）
   - 结构化结果生成
   - 可视化数据准备

**边界约束**：
- 不处理图像输入
- 不进行 OCR 相关操作
- 输入必须是经过 I1+I2 处理的标准化文本

### 3.5 三层协作原则

1. **单向依赖**：I2 依赖 I1，Analyze 依赖 I2，禁止反向依赖
2. **接口隔离**：每层通过标准化接口通信，内部实现可独立演进
3. **错误传播**：下层错误必须明确类型，上层决定处理策略
4. **数据溯源**：每层输出需携带来源标记，便于问题追踪

## 4. I1 最低标准输出协议

### 4.1 核心数据模型

```python
class I1Output:
    """I1 图转文节点标准输出模型"""
    
    # 基础字段
    success: bool                      # 是否成功
    error_code: Optional[str]          # 错误码（success=False时）
    error_message: Optional[str]       # 错误描述
    
    # 成功时输出
    text: str                          # 转写文本（含原始换行）
    confidence: float                  # 整体置信度 0-1
    language: str                      # 识别语言（'zh', 'en', 'mixed'）
    
    # 结构化信息
    segments: List[Segment]            # 文本片段详情
    
    # 元数据
    processing_time_ms: int            # 处理耗时
    image_hash: str                    # 图像哈希（用于去重）
    model_version: str                 # OCR 模型版本

class Segment:
    """文本片段详情"""
    
    text: str                         # 片段文本
    confidence: float                 # 片段置信度
    bbox: List[int]                   # 边界框 [x1, y1, x2, y2]
    is_speaker_marker: bool           # 是否为说话人标记
    speaker_label: Optional[str]      # 说话人标签（如'A', 'B'）
```

### 4.2 接口定义

```python
# RESTful 接口
POST /api/v1/image-to-text

# 请求
{
    "image_base64": "data:image/png;base64,...",  # Base64编码图像
    "language_hint": "zh",                        # 语言提示（可选）
    "require_segments": true                      # 是否要求分段输出（可选）
}

# 成功响应（HTTP 200）
{
    "success": true,
    "data": {
        "text": "A: 你好吗？\nB: 我很好，谢谢！",
        "confidence": 0.92,
        "language": "zh",
        "segments": [
            {
                "text": "A:",
                "confidence": 0.98,
                "bbox": [10, 20, 30, 40],
                "is_speaker_marker": true,
                "speaker_label": "A"
            },
            {
                "text": "你好吗？",
                "confidence": 0.95,
                "bbox": [35, 20, 120, 40],
                "is_speaker_marker": false,
                "speaker_label": null
            }
            # ... 更多片段
        ],
        "processing_time_ms": 325,
        "image_hash": "a1b2c3d4",
        "model_version": "ocr-v1.2.0"
    }
}

# 失败响应（HTTP 400/500）
{
    "success": false,
    "error_code": "IMAGE_QUALITY_TOO_LOW",
    "error_message": "图像质量过低，无法识别文字",
    "suggestions": ["请上传更清晰的截图", "尝试裁剪文字密集区域"]
}
```

### 4.3 质量指标要求

| 指标 | 最低要求 | 目标要求 | 测量方法 |
|------|----------|----------|----------|
| 中文准确率 | ≥85% | ≥95% | 字符级比对 |
| 处理时间 | ≤5秒 | ≤2秒 | P95 响应时间 |
| 可用性 | ≥99% | ≥99.9% | 月度统计 |
| 错误分类率 | ≥90% | ≥95% | 人工审核 |

### 4.4 错误码体系

| 错误码 | 含义 | 处理建议 |
|--------|------|----------|
| IMAGE_FORMAT_UNSUPPORTED | 图像格式不支持 | 转换格式后重试 |
| IMAGE_QUALITY_TOO_LOW | 图像质量过低 | 上传更清晰图像 |
| NO_TEXT_DETECTED | 未检测到文字 | 确认图像包含文字 |
| LANGUAGE_NOT_SUPPORTED | 语言不支持 | 切换语言提示 |
| SERVICE_UNAVAILABLE | 服务暂时不可用 | 稍后重试 |

## 5. 新旧输入链路对比

### 5.1 V3.5 及之前输入链路

```
传统链路（单一路径）：
用户文本输入 → [Analyze 分析引擎] → 结果

特点：
- 仅支持纯文本输入
- 输入质量依赖用户
- 无预处理/标准化层
- 分析引擎承担输入验证
```

### 5.2 V3.6 新输入链路

```
双模输入链路：

路径1（文本输入）：
用户文本输入 → [I2 清洗层] → [Analyze 分析引擎] → 结果

路径2（图像输入）：
用户图像输入 → [I1 图转文层] → [I2 清洗层] → [Analyze 分析引擎] → 结果

特点：
- 双模输入（文本+图像）
- 分层处理（I1+I2+Analyze）
- 输入标准化前置
- 分析引擎专注核心业务
- 用户确认环节保障质量
```

### 5.3 架构改进点

1. **关注点分离**：输入处理与情感分析解耦
2. **质量前置**：输入质量问题在分析前暴露
3. **扩展性**：新增输入模式不影响分析引擎
4. **可维护性**：各层可独立升级优化
5. **用户体验**：提供图像输入便捷通道

## 6. 与 Analyze 引擎的衔接规范

### 6.1 核心原则：图转文接口独立

1. **物理分离**：I1 服务可独立部署，与 Analyze 引擎无直接依赖
2. **协议独立**：I1 输出协议专为图转文设计，不耦合分析业务
3. **版本独立**：I1 与 Analyze 可独立进行版本升级
4. **故障隔离**：I1 故障不影响文本输入路径及 Analyze 核心功能

### 6.2 必经之路：所有图转文结果先入 I2 确认层

```
强制流程：
I1 输出 → 前端展示 → 用户确认/编辑 → I2 清洗 → Analyze

设计理由：
1. **准确性保障**：OCR 准确率非 100%，必须人工确认
2. **标准化入口**：确保 Analyze 接收统一格式输入
3. **数据质量**：用户修正数据可用于 OCR 模型优化
4. **用户体验**：给予用户控制感，提升结果可信度
```

### 6.3 衔接接口定义

```python
# I1 → 前端（用于展示确认）
POST /api/v1/image-to-text
# 返回 I1Output，前端展示 text 并允许编辑

# 前端 → I2（用户确认后清洗）
POST /api/v1/text/clean
{
    "raw_text": "A: 你好吗?\nB: 我很好!",  # 用户确认/编辑后的文本
    "source": "ocr",
    "original_ocr_text": "A: 你好吗?\nB: 我很好!"  # 原始 OCR 文本（可选）
}

# I2 → Analyze（标准化后分析）
# 使用现有 /analyze 接口，传入 cleaned_text
```

### 6.4 错误处理衔接

| 故障点 | 影响范围 | 降级方案 |
|--------|----------|----------|
| I1 服务不可用 | 图像输入路径 | 降级为手动文本输入 |
| I1 识别质量低 | 单次图像输入 | 用户手动编辑修正 |
| I2 清洗失败 | 单次输入 | 返回具体原因，前端提示 |
| Analyze 引擎故障 | 所有输入 | 现有兜底机制 |

## 7. 后续分 Phase 推进方式

### Phase 1（当前）：架构冻结与接口定义
- **目标**：完成本架构文档，冻结 I1 节点设计
- **产出**：标准化接口协议、质量指标、错误码体系
- **不实现**：具体 OCR 技术实现

### Phase 2：Mock 服务与前端集成
- **目标**：实现 I1 Mock 服务，前端完成双模输入界面
- **产出**：可演示的双模输入流程
- **技术**：I1 Mock 返回固定文本，验证全链路通畅

### Phase 3：OCR 服务技术选型与接入
- **目标**：接入真实 OCR 服务，达到最低质量指标
- **产出**：生产可用的 I1 服务 v1.0
- **技术选项**：云端 OCR API（百度/腾讯/阿里）、开源 OCR 引擎、混合方案

### Phase 4：性能优化与准确率提升
- **目标**：优化处理性能，提升识别准确率至目标要求
- **产出**：I1 服务 v2.0，满足生产 SLA
- **措施**：图像预处理优化、模型调优、用户反馈数据利用

### Phase 5：高级功能扩展
- **目标**：扩展输入场景覆盖，提升用户体验
- **产出**：I1 服务 v3.0，支持批量处理、历史记录等
- **功能**：多图关联、智能分段、置信度可视化

## 8. 本阶段明确不做事项

### 8.1 技术实现绑定
1. **不绑定**具体 OCR 服务提供商
2. **不绑定**本地或云端部署模式
3. **不绑定**特定深度学习框架
4. **不绑定**图像预处理算法细节

### 8.2 高级功能
1. **不实现**实时视频流 OCR
2. **不实现**手写体识别
3. **不实现**复杂版面分析（表格、多栏）
4. **不实现**多语言混合识别（除中英文）

### 8.3 业务逻辑
1. **不添加**情感分析相关逻辑
2. **不添加**用户身份识别
3. **不添加**内容过滤策略
4. **不添加**个性化适配

### 8.4 性能与规模
1. **不要求**毫秒级响应
2. **不支持**大规模批量处理（>10张/次）
3. **不保证**离线可用性
4. **不实现**边缘设备部署

## 9. 技术实现建议（非约束性）

### 9.1 推荐技术栈选项

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 云端 OCR API | 准确率高、维护简单 | 网络依赖、成本敏感 | 快速上线、中小规模 |
| 开源 OCR 引擎 | 可控性强、成本低 | 准确率需调优、维护复杂 | 数据敏感、大规模 |
| 混合方案 | 灵活平衡 | 架构复杂 | 高可用性要求 |

### 9.2 架构部署建议

1. **微服务化部署**：I1 作为独立微服务，RESTful 接口暴露
2. **异步处理**：支持同步/异步两种调用模式
3. **缓存机制**：相同图像哈希可缓存识别结果
4. **监控埋点**：关键质量指标实时监控

### 9.3 演进路线图建议

1. **短期**：云端 API 快速验证
2. **中期**：引入开源引擎，降低成本
3. **长期**：自研优化模型，形成技术壁垒

| 术语 | 全称 | 说明 |
