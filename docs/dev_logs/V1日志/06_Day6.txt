## 三、今日核心工作内容

### 3.1 Coze Workflow API 正式接入

完成后端对 Coze Workflow 的真实调用，系统链路变为：

```text
用户输入
→ Streamlit 前端
→ FastAPI 后端
→ Coze Workflow API
→ 返回分析结果
→ 前端展示
```

本阶段完成了以下关键动作：

* 校正 workflow_id 配置方式
* 明确调用对象必须为已发布工作流
* 在后端加入 payload、headers、timeout 等完整请求逻辑
* 接入真实返回结果而非本地 mock 数据

---

### 3.2 结束节点输出问题定位与修复

本阶段定位到一个关键工程问题：

```text
工作流已执行成功
≠ API 一定返回正确结果
```

问题根源在于：

* 结束节点输出模式配置不当
* 返回文本 / 返回变量混用
* 输出变量类型配置错误
* 最终输出未正确绑定上游节点结果

经过排查，最终明确：

* 结束节点应采用 **返回变量**
* 输出变量应直接绑定最终结果变量
* 不应把最终结果作为普通文本模板返回

这一轮排查后，工作流“执行成功但前端拿到空结果”的问题被定位清楚。

---

### 3.3 后端异常处理与调试能力补强

为了定位 Coze 调用问题，后端增加了完整调试输出与兜底机制，包括：

* 打印 HTTP status code
* 打印原始 response text
* 打印解析后的 JSON 结果
* 捕获请求异常
* 在失败时返回标准兜底 JSON，避免前端崩溃

这一步非常关键，它让系统从“黑盒调试”变成了“可观察调试”。

---

## 四、关键问题与修复过程

### 4.1 workflow_id 配置错误

早期版本中，workflow_id 使用了自定义名称，而不是实际可调用的工作流标识，导致：

* 请求看似发出
* 实际没有命中目标工作流
* 返回结果为空或不符合预期

修复后：

* 使用正确的 workflow_id
* 明确区分“工作流名称”与“API 调用 ID”

---

### 4.2 请求已到达 Coze，但执行超时

在 workflow_id 修正后，后端日志显示：

```text
HTTPSConnectionPool(host='api.coze.cn', port=443): Read timed out
```

这说明：

* 请求已经成功发到 Coze
* API 地址和鉴权基本有效
* 真正的问题不是“没调通”
* 而是**工作流执行时间过长**

也就是说，系统已经从“接入问题”进入“性能问题”阶段。

---

### 4.3 模型与节点链路过重，导致整体超时

在使用较重模型时，S1–S6 六节点链路总耗时过高，尤其：

* S5（策略建议生成）耗时显著
* S6（结果封装）再次调用模型，形成重复推理

由此暴露出两个关键问题：

1. 节点职责划分过重
2. 输出封装层不应继续使用大模型做二次总结

---

### 4.4 轻量模型替换后，速度显著提升，但质量下降

为解决超时问题，本阶段尝试将多个节点切换为更轻量模型。结果显示：

* 响应速度显著提升
* 全链路执行时间明显缩短
* 但分析质量下降明显
* S2 输出大量“无法判断”
* 导致 S3、S4、S5 连锁失效

这表明：

```text
速度优化成功
但模型能力与任务复杂度出现错配
```

---

## 五、关键认知沉淀（本阶段最重要）

### 5.1 不是所有节点都适合轻模型

本阶段最大的工程认知之一是：

```text
轻模型适合做“格式化与提取”
强模型适合做“推理与决策”
```

也就是说：

| 节点        | 更适合的能力类型     |
| --------- | ------------ |
| S1 文本预处理  | 轻模型          |
| S2 信息提取   | 轻模型（但需降规则强度） |
| S3 情绪趋势判断 | 中/强模型        |
| S4 关系阶段判断 | 强模型          |
| S5 策略建议生成 | 强模型          |

---

### 5.2 S2 是整个系统的数据源，不能“全无法判断”

通过实际输出发现：

```json
{
  "initiative": "无法判断",
  "response_speed": "无法判断",
  "response_length": "无法判断",
  "emotional_tone": "无法判断",
  "topic_depth": "无法判断",
  "key_signals": []
}
```

这说明不是后续判断模块的问题，而是：

```text
S2 没有产出有效信号
→ 后续模块全部失去输入基础
```

因此本阶段明确了一个关键原则：

```text
S2 必须允许“弱判断”
不能轻易把低信息样本全部归为“无法判断”
```

---

### 5.3 S6 不应再做一次模型推理

本阶段最重要的架构修正之一是：

```text
S6 不应是“再思考一次”
S6 应该只是“结果组织层”
```

因为 S5 已经承担了核心策略生成任务，如果 S6 再调用模型，会产生三个后果：

* 增加耗时
* 稀释信息
* 降低结果质量

因此，Day 6 的关键优化方向已经明确：

> **删除或弱化 S6 的模型属性，让结束节点直接返回核心结果**
