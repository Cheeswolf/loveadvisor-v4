## 四、关键问题记录（重点）

### ❗问题：LLM Token 上下文超限

#### 4.1 问题表现

在使用 Claude Code 执行任务时，出现错误：

```
maximum context length is 131072 tokens
requested 131467 tokens
```

---

### 4.2 错误原因分析

问题本质为：

| 项目             | token     |
| -------------- | --------- |
| 历史消息（messages） | ≈ 99k     |
| completion 参数  | 32000     |
| 总计             | 超过 131072 |

---

### 4.3 关键结论

该问题：

* ❌ 与余额无关
* ❌ 与网络无关
* ❌ 与 API 调用失败无关

本质是：

> ❗模型上下文窗口被撑爆（Context Overflow）

---

### 4.4 根本原因

开发过程中使用了：

* 长对话历史（项目背景 + 多轮讨论）
* 超大 completion 参数（32000）
* 单次任务输入过大

导致：

> “聊天式开发”触发 token 极限

---

## 五、工程级修复方案（重要）

### 5.1 开发模式调整

从：

```
聊天式开发（大上下文）
```

转为：

```
任务分片式开发（小上下文）
```

---

### 5.2 Claude Code 调用规范

#### 强制规则：

1. 每次只提交“最小任务单”
2. 禁止传入完整项目背景
3. 禁止长历史对话复用
4. 必须使用新对话执行关键任务

---

### 5.3 Token 控制策略

| 项目         | 原设置   | 新规范            |
| ---------- | ----- | -------------- |
| completion | 32000 | ≤ 2000         |
| 输入规模       | 无限制   | 单任务 ≤ 2k token |
| 对话长度       | 累积    | 分阶段隔离          |

---

### 5.4 开发补丁（流程级）

引入：

> **「任务分片执行机制」**

执行方式：

```
Phase → 拆解 → 单任务 → Claude Code执行 → 验收
```

---

### 5.5 工程影响评估

| 维度      | 影响     |
| ------- | ------ |
| 开发效率    | ↑ 提升   |
| 稳定性     | ↑ 提升   |
| 可控性     | ↑ 显著提升 |
| token风险 | ↓ 消除   |
