## 四、调试过程与关键问题定位

本日虽然最终完成了批测系统落地，但实际过程中并非“一次跑通”，而是经历了一轮非常典型且有价值的工程排障过程。

---

### 4.1 Python 环境问题暴露并修复

最初在 PowerShell 中直接运行：

```powershell
python test_system/scripts/batch_test.py
```

系统提示无法找到 `python.exe`。排查后确认：

* Python 本体实际已安装
* 但当前 PowerShell 会话优先命中了 `WindowsApps` 下的 `python.exe`
* 当前终端未正确拿到可用 Python 路径

最终通过：

* 显式指定 Python 路径验证解释器可用
* 临时修正当前 PowerShell 的 `PATH`
* 成功恢复 `python --version` 与 `python test_system/scripts/batch_test.py` 的正常执行

这一问题解决后，Day 2 的批测脚本终于具备了运行条件。

---

### 4.2 批测系统首次运行后出现“全体无法判断”异常

环境恢复后，`batch_test.py` 虽然可以跑完 15 条样本并正常生成结果文件，但首轮输出出现了明显异常：

* A / B / C / D 组样本几乎全部输出为 `无法判断 / 无法判断`
* 只有 E 组显示“看似正确”

这说明问题不在批测脚本本身，而在后端主链路。也就是说：

```text
批测系统是通的
但 /analyze 并没有真正输出有效业务结果
```

这一步非常关键，它证明自动评测系统已经具备了“反向发现后端故障”的能力。

---

### 4.3 通过日志逐层定位 `/analyze` 主链路问题

为定位问题来源，本阶段对以下位置逐步加日志：

* `app/api/analyze.py` 路由入口
* `app/services/pipeline_service.py` 文件加载与函数入口
* pipeline 内部 S2 / S3 / R1 / S5 / O1 各关键步骤

排查过程中，先后识别出以下问题：

#### 问题一：路由与 service 虽加载成功，但 pipeline 调用签名不一致

初始状态下：

* `analyze.py` 中调用的是 `run_pipeline(text)`
* 但 `pipeline_service.py` 中定义为 `run_pipeline(text, user_question)`

导致请求实际未正常进入主逻辑，直接抛出参数缺失错误。

---

#### 问题二：V2.5 重构后遗留变量名不一致

修复签名问题后，又发现 `pipeline_service.py` 内部仍混用了旧变量名：

* `text`
* `chat_text`

以及调试变量名不一致：

* 实际变量为 `s2 / s3`
* 但打印时仍写成 `s2_output / s3_output`

同时 `user_question` 在 pipeline 中仍被引用，但当前接口和批测体系并未提供该字段。

这一轮说明：当前工程结构虽已完成拆分，但服务层代码中仍保留了明显的 V1/V2 过渡遗留。

---

#### 问题三：文件顶层调试打印误用函数内部变量

在调试 `pipeline_service.py` 时，曾将：

```python
print("=== PIPELINE INPUT ===", text)
```

误放到文件顶层，导致模块导入阶段直接报 `NameError`。后续修正为：

* 文件顶层仅打印 `FILE LOADED`
* 函数内部再打印 `PIPELINE INPUT`

这一问题虽然属于调试失误，但反过来也验证了：当前运行的确实是新项目结构中的目标文件。

---

### 4.4 `/analyze` 主链路最终打通

经过上述几轮修复后，使用单样本 A1 进行接口测试时，后端日志首次完整输出了以下链路：

* `ANALYZE ROUTE HIT`
* `PIPELINE FUNCTION HIT`
* `AFTER EXTRACT WORKFLOW`
* `DEBUG S2`
* `DEBUG S3`
* `DEBUG R1`
* `AFTER S5 WORKFLOW`
* `DEBUG S5`
* `FINAL OUTPUT`

其中关键结果为：

* S2 正常提取基础互动信号
* S3 正常提取高级关系信号
* R1 正常输出 `relationship_stage` 与 `interest_level`
* S5 正常生成心理分析、风险点、建议与 next_step
* O1 正常组装最终输出
* `/analyze` 返回 `200 OK`

这标志着：V2.5 的自动评测系统不再是在“打假链路”，而是开始真正测到业务结果。
