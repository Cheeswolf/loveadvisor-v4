# 📘 LoveAdvisor V2.5 项目开发日志（临时中止记录）

## 一、事件性质

本次不是常规规则误判，也不是后端链路崩溃，而是**外部依赖失效导致的强制中止**。

经临时排查脚本验证：

* `EXTRACT_WORKFLOW_ID` 与 `S5_WORKFLOW_ID` 均能正常发起请求；
* HTTP 状态码为 `200`；
* 但返回体中业务码为 `4028`；
* 提示内容为：**免费额度已用尽，需升级付费计划**；
* 因此 `run_workflow()` 实际拿不到可用业务数据，只能返回空结果或调试错误结构。

这意味着当前问题不属于本地规则层，也不属于批测脚本本身，而是：

```text
外部 Coze Workflow 配额耗尽
→ S2 / S3 / S5 无法继续返回真实结果
→ /analyze 失去上游输入
→ 批测结果整体失真
```

---

## 二、今日中止原因

### 1. 直接原因

Coze API 返回：

* `code = 4028`
* `msg = Your free quota has been used up...`

说明当前账号下用于 Workflow API 的免费额度已耗尽。

### 2. 表层表现

项目内表现为：

* `EXTRACT RESULT = {}`
* `S5 RESULT = {}`
* `S2 = {}`
* `S3 = {}`
* `R1` 因拿不到有效输入而统一输出“无法判断 / 无法判断”

### 3. 为什么必须中止

当前若继续沿用 `/analyze → Coze → S2/S3/S5` 的主链路开展批测，会出现两个严重问题：

* 所有批测结果都将被“外部空返回”污染；
* 无法再客观评估 `rules.py`、`interest_level`、`next_step` 的真实优化效果。

因此，继续当前开发方式将失去评测意义，必须立即中止现行链路下的迭代。

---

## 三、当前阶段成果保全

虽然项目被迫中止，但截至中止前，以下成果已经成立，且需要被正式保留：

### 1. V2.5 评测体系已经落地

项目已完成：

* 测试集结构化；
* 批量评测脚本；
* 自动结果落表；
* 核心指标统计；
* 错误分类机制。

### 2. 项目主架构已完成工程化拆分

当前工程结构已经从单文件演进为：

* `app / api / services / rules / utils / frontend / test_system` 分层结构；
* `/analyze` 仅作为入口；
* `pipeline_service` 负责主流程；
* `rules.py` 负责规则判定；
* `output_builder.py` 负责 O1 收口。

### 3. V2.5 的开发方法论已经明确

项目已正式进入：

> **评测驱动的稳定化开发阶段**
> 重点是“测试工程化、规则稳定化、建议收敛化”，而不是继续盲目扩链路或补 prompt。

### 4. 当前中止不是架构失败

此次中止是外部 API 配额问题，并不推翻以下结论：

* 本地规则层仍具有独立优化价值；
* 自动评测体系本身有效；
* 后端工程结构已具备继续开发条件；
* 问题集中在外部模型依赖而非本地代码组织。

---

## 四、当前中止结论

本项目即日起进入：

> **临时强制中止状态（External Dependency Suspension）**

中止原因定义为：

```text
外部 Workflow 配额耗尽，导致真实业务链路失效，
当前批测结果不再具备优化参考价值。
```

当前不再继续：

* 基于 `/analyze` 的全量批测；
* 基于 Coze 返回结果的规则精修；
* 基于 S5 当前在线结果的表达收敛工作。

---

# 📌 临时新方案（重组版）

下面这部分不是长期终版，而是**临时组织方案**，目的是把项目从“被外部额度卡死”拉回到“可继续推进”的状态。

---

## 一、临时阶段命名

建议将接下来这段时间命名为：

> **LoveAdvisor V2.5-P：脱外部依赖的本地稳定化过渡阶段**

这个阶段不是重做项目，而是：

* 暂停外部 Coze 依赖；
* 保住规则层与评测层成果；
* 建立离线可继续运行的最小开发闭环。

---

## 二、临时重组目标

### G1：冻结当前 Coze 链路

不再把 Coze 当作当前阶段的可用开发依赖。

### G2：转入离线规则开发

后续 `rules.py`、`interest_level`、`next_step_action` 的调试，不再依赖在线 workflow。

### G3：建立 Mock / 固定样本驱动机制

用手工冻结的 `S2_output / S3_output` 做本地规则回归。

### G4：保留自动评测体系

评测系统不废弃，但临时改为“离线输入模式”。

---

## 三、临时技术路线

### 路线 A：业务主链路暂时冻结

当前这条链路：

```text
前端输入
→ /analyze
→ Coze Workflow
→ S2 / S3
→ R1
→ S5
→ O1
→ 前端展示
```

暂时停止作为“开发测试主路径”。

原因是上游不可用，继续跑只会污染结论。

---

### 路线 B：切换为离线规则闭环

临时开发主路径改为：

```text
固定测试样本
→ 手工冻结 S2 / S3
→ R1（rules.py）
→ next_step_action
→ O1 映射
→ 指标验证
```

也就是：

* 不再依赖在线 workflow；
* 先让本地规则层继续进化；
* 等额度恢复后，再重新接回 Coze。

---

### 路线 C：S5 暂时降级为非核心

当前阶段内：

* `psychological_analysis`
* `risk_points`
* `suggestions`

可以暂时不作为主要优化对象；

真正保留优化重点的是：

* `relationship_stage`
* `interest_level`
* `next_step_action`

因为这三项才是规则核心，也是当前最可控的部分。

---

### 第 1 步：冻结现状

立即保留以下文件与状态：

* 当前 `rules.py`
* 当前 `batch_test.py`

* 当前 `metrics_report.json`
* 当前 `coze_client.py` 的调试增强版

目的：确保后续恢复时有清晰基线。

作用：

### 第 3 步：建立离线测试集

将现有 15 条测试样本继续保留，但新增一份配套的：


每条样本包含：

* `sample_id`
* `s2_output`
* `s3_output`
* `expected_relationship_stage`
* `expected_interest_level`
* `expected_next_step_action`

这样就能绕过 Coze，直接验证规则层。

---

### 第 4 步：恢复 Day 5 任务，但改为离线版

原本 Day 5 的任务是：

> interest_level 分档专项修复

这个目标不变，但执行路径改为：

* 只测 `rules.py`
* 不再调用 `/analyze`
* 不再依赖 S2/S3 在线返回

---

### 第 5 步：等待条件恢复后，再重新接回 Coze

恢复条件包括任一：

* 充值 Coze API 可用额度；
* 更换可用账号 / token；
* 替换为其他模型服务；
* 或把 S2/S3/S5 局部迁移到本地或其他接口。

在此之前，不再把 Coze 作为主开发环境。

---

## 五、临时阶段验收标准

这个过渡阶段不追求“完整产品可用”，只追求：

1. 规则层仍可继续开发；
2. 自动评测体系不废弃；
3. 项目不因外部额度中断而停摆；
4. 后续可无缝切回真实链路。

## 六、最终判断

这次“强制中止”是合理的，原因充分，而且是工程上正确的决定。

因为当前若不停止，所有输出都会被外部空返回污染，继续开发只会制造假结果。
但这次中止**不代表项目失败**，反而证明了：

* 你已经把问题从“模糊怀疑”推进到了“精确定位”；
* 已经能够区分“代码问题”和“外部依赖问题”；
* 项目已经具备从在线依赖切换到离线规则开发的基础。
