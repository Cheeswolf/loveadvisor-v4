因此，Day 4 的目标不再是继续补 Prompt，而是正式将高层关系判断后端化。

### 3.2 relationship_stage 判定逻辑正式后端化

今日正式完成 `relationship_stage` 的规则化实现，采用固定优先级判定路径：

```text id="nfzxj"
无法判断
→ 暧昧期
→ 拉扯期
→ 冷淡期
→ 初识期
```

其中，核心逻辑如下：

#### ① 无法判断

当前置信息不足、互动结构不足、S2 多个核心字段为“无法判断”、且 S3 没有明显高级关系信号时，优先输出：

```text id="t0jj9"
无法判断
```

该规则用于拦截信息极少场景，避免系统继续犯“该不判却强判”的错误。

#### ② 暧昧期

满足以下任一结构时，输出：

```text id="kibw0"
暧昧期
```

* 存在明确亲密拉近信号，且没有明显拒绝
* 存在关系试探，同时对方给出正向承接，且没有明显后退

#### ③ 拉扯期

满足以下任一结构时，输出：

```text id="r8pbk"
拉扯期
```

* 存在推进与后退并存的 push-pull 结构
* 存在 intimacy 与 rejection 同时出现
* 存在 relationship_probe 与 rejection 同时出现

#### ④ 冷淡期

满足以下任一结构时，输出：

```text id="q9f83"
冷淡期
```

* 存在持续低投入
* 明确拒绝/回避，且基础情绪色调偏冷
* 明显回避，且整体互动偏冷

#### ⑤ 初识期

在不存在 intimacy / push-pull / sustained_coldness 的前提下，作为正常互动场景的兜底输出：

```text id="u7tgp"
初识期
```

这一规则层落地后，高层阶段判断首次变成了可读、可改、可测的显式后端逻辑。

---

### 3.3 interest_level 判定逻辑正式后端化

今日同步完成 `interest_level` 的规则化实现，并明确其含义不是“谁更喜欢谁”，而是：

```text id="csfke"
关系推进能力
```

当前实现逻辑如下：

* 若 `relationship_stage = 无法判断` → `interest_level = 无法判断`
* 若处于暧昧期，且存在正向承接、无拒绝 → `高`
* 若处于暧昧期 → `中`
* 若处于拉扯期 → `中`
* 若处于初识期 → `低`
* 若处于冷淡期 → `低`

同时补入联动规则，避免出现“阶段不可判，但兴趣等级被强行打低”或“存在明显后退，却仍然输出高兴趣”的逻辑冲突。

---

### 3.4 backend 链路完成 R1 接入

今日在后端正式完成以下链路打通：

```text id="3i7q4"
Coze Workflow
→ 返回 S2_output / S3_output
→ backend 解析
→ infer_r1(s2, s3)
→ 返回 R1_output
```

在接入过程中，先后解决了两个关键工程问题：

#### 问题一：Coze 执行成功，但 `data` 为空

最初接口返回 `code = 0` 但 `data = ""`，说明工作流运行成功，但结束节点没有正确返回变量。后经调整结束节点输出方式，确保 S2_output / S3_output 作为变量被真正返回，问题得到修复。

#### 问题二：`data` 不为 dict，而是字符串形式的 JSON

在结束节点修正后，Coze 已开始返回内容，但 `raw_coze_response["data"]` 并非字典，而是字符串形式的 JSON，对应内容中又嵌套了字符串版 `S2_output` / `S3_output`。为解决这一问题，后端补充了对多层 JSON 字符串的安全解析逻辑，最终成功提取出有效的 S2 / S3 结果，并顺利完成 R1 推理。

这一步完成后，Day 4 的核心工程链路正式打通。
