# 📘 LoveAdvisor V2 项目开发日志 — Day 1（架构重整与协议设计）

## 一、今日目标

完成 LoveAdvisor V2 的第一阶段重构工作，明确系统级升级方向，输出可执行的架构与协议方案。

本日目标包括：

* 重整 V2 总体架构
* 删除 V1 中不合理的高层模型判断链路
* 明确各节点职责边界
* 统一输入输出协议
* 为后端规则引擎与输出组装层预留接口

实现目标：

> 将系统从“LLM 串联式工作流”升级为“职责清晰、规则可控、协议统一”的可重构工程体系。 

---

## 二、阶段背景

V1 已完成工作流打通、前后端分离、真实 API 调用与基础产品闭环，但也暴露出明显问题：

* S2 作为上游信号源，输出过于保守，导致后续链路连锁失效
* S4 直接由模型输出关系阶段，稳定性不足
* S6 存在二次推理，增加耗时并稀释信息
* 多个强模型节点叠加，造成整体性能压力与超时风险

因此，V2 不再继续以“补 Prompt”为核心，而是正式进入结构性重构阶段。  

---

## 三、今日核心成果

## 3.1 确立 V2 核心原则

今日正式确认 V2 的核心设计原则：

```text
模型负责提信号
规则系统负责下结论
生成层负责把结论变成人话
```

这标志着项目逻辑从“模型直接判断”升级为“模型提取 + 后端规则 + 输出组装”的新范式。 

---

## 3.2 重构总体架构

今日确定 V2 推荐架构如下：

```text
用户输入
→ S1 文本清洗
→ S2 基础互动信号提取
→ S3 高级关系信号提取
→ R1 后端规则引擎
→ S5 策略建议生成
→ O1 输出组装层
→ 前端展示
```

相较 V1，完成了以下关键调整：

| 模块 | V1       | V2          |
| -- | -------- | ----------- |
| S1 | 文本清洗     | 保留          |
| S2 | 提取 + 弱判断 | 仅保留基础互动信号提取 |
| S3 | 情绪趋势判断   | 改为高级关系信号提取  |
| S4 | 模型判阶段    | 删除          |
| R1 | 无        | 新增后端规则引擎    |
| S5 | 建议生成     | 保留，改输入来源    |
| S6 | 输出封装     | 删除推理属性      |
| O1 | 无        | 新增输出组装层     |

这一轮调整的核心，是把高层关系标签从模型节点中剥离出来，转交给可控的后端规则系统。 

---

## 3.3 完成节点职责冻结

为避免后续再次出现“提取、判断、生成、封装混杂”的问题，今日完成各节点职责边界定义：

| 节点 | 职责                                       |
| -- | ---------------------------------------- |
| S1 | 清洗并标准化聊天文本                               |
| S2 | 提取基础互动行为信号                               |
| S3 | 提取高级关系特征                                 |
| R1 | 规则判定 relationship_stage / interest_level |
| S5 | 生成心理分析、风险与建议                             |
| O1 | 统一整合最终输出                                 |

同时明确：

* S1 不做判断
* S2 不输出关系阶段
* S3 不输出 relationship_stage
* R1 不生成建议文案
* O1 不再进行模型推理

这一步完成后，V2 的节点边界首次具备明确工程约束。 

---

## 四、协议设计成果

## 4.1 S1 协议确定

S1 明确为“关系分析预处理器”，职责包括：

* 统一说话人格式
* 保留时间顺序
* 去除系统噪音
* 保留“哈哈、嗯、哦”等低信息情绪内容
* 压缩无效文本长度

输出形式明确为标准化文本，而非 JSON。
同时明确禁止输出 reasoning。 

---

## 4.2 S2 协议确定

S2 今日完成字段级协议冻结，输出结构确定为：

```json
{
  "initiative": "",
  "response_length": "",
  "emotional_tone": "",
  "topic_depth": "",
  "interaction_reciprocity": "",
  "key_signals": []
}
```

并明确：

* 只输出低层、可观测信号
* 不直接输出“暧昧期 / 拉扯期 / 冷淡期”等高层标签
* `key_signals` 限制为 2–4 条
* 输出必须为严格 JSON

这一步是对 V1 中 S2 过重、过保守问题的正式修复起点。 

---

## 4.3 S3 协议确定

S3 被重新定义为“高级关系信号提取层”，输出结构确定为：

```json
{
  "has_intimacy_signal": false,
  "has_relationship_probe": false,
  "has_positive_reciprocity": false,
  "has_rejection_signal": false,
  "has_push_pull_pattern": false,
  "has_sustained_coldness": false,
  "signal_summary": []
}
```

今日明确：

* S3 只提取高级关系特征
* 不直接判断 relationship_stage
* 设计目的是降低模型直接判阶段的难度和波动性

这使得高层结论第一次具备了中间可观测特征支撑。 

---

## 4.4 R1 接口定义完成

今日完成 R1 后端规则引擎的职责与接口预定义：

```json
{
  "relationship_stage": "",
  "interest_level": ""
}
```

并明确其核心任务：

* 根据 S2 + S3 的结构化信号进行规则判定
* 高层关系标签后端化
* 所有规则集中管理、可测试、可复盘

这一步是 V2 最关键的架构升级，也是整个系统从“Prompt 项目”走向“规则增强系统”的标志。 

---

## 4.5 S5 与 O1 协议完成重定义

### S5

S5 今日被重新定义为：

* 输入：S2_output、S3_output、R1_output、user_question
* 输出：psychological_analysis、risk_points、suggestions、next_step

并明确：

* 建议生成必须依赖结构化信号与规则结果
* 不得推翻上游阶段结论
* 不再承担阶段判断职责

### O1

O1 今日被定义为最终输出整合层：

* 不使用 LLM
* 后端直接组装
* 缺失字段统一兜底
* 只做映射与整合，不做推理

这一步正式替代了 V1 中存在冗余问题的 S6。
