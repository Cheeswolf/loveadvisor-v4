# 📘 LoveAdvisor V2 项目开发日志 — Day 3（S3 新建与高级关系信号层验收）

## 一、今日目标

完成 LoveAdvisor V2 第三阶段重构工作，正式新建 **S3：高级关系信号提取层**，并围绕边界样本完成首轮调试、补丁迭代与最终验收。

本日目标包括：

* 新建 S3 节点
* 冻结 S3 输出协议
* 完成 S3 Prompt 初版设计
* 解决高级关系信号识别中的关键边界误判
* 用重点样本完成最终验收

实现目标：

> 将系统从“只有基础互动信号层”的状态，升级为“基础互动信号 + 高级关系信号”的双层结构，为 Day 4 的 R1 后端规则引擎提供稳定输入。  

---

## 二、阶段背景

在 V2 Day 2 完成 S2 重构后，系统已经具备稳定输出基础互动信号的能力，但仍缺少一层专门负责提取“关系相关高级特征”的中间层。按照 V2 正式实施方案，Day 3 的核心任务不是让模型直接输出 relationship_stage，而是先把更可观测、更可复用的高级关系信号独立出来，例如：

* 是否存在亲密拉近信号
* 是否存在关系试探
* 是否存在正向承接
* 是否存在回避或拒绝
* 是否存在推进与后退并存的拉扯结构
* 是否存在持续性冷淡

这一步的本质，是把高层关系判断进一步拆解，降低模型直接判阶段的难度与波动性。

---

## 三、今日核心成果

### 3.1 完成 S3 节点新建

今日正式将 S3 定义为：

```text
高级关系信号提取层
```

其职责明确为：

* 识别高级关系特征
* 输出布尔型结构化信号
* 不直接输出 relationship_stage
* 不直接输出 interest_level
* 不承担建议生成职责

这意味着 V2 的职责分层进一步被压实，S3 成为连接 S2 与 R1 的关键中间层。

---

### 3.2 冻结 S3 输出协议

今日完成 S3 输出协议冻结，最终结构为：

```json
{
  "has_intimacy_signal": false,
  "has_relationship_probe": false,
  "has_positive_reciprocity": false,
  "has_rejection_signal": false,
  "has_push_pull_pattern": false,
  "has_sustained_coldness": false,
  "signal_summary": []
}
```

字段职责明确如下：

| 字段                       | 含义              |
| ------------------------ | --------------- |
| has_intimacy_signal      | 是否出现明确亲密拉近表达    |
| has_relationship_probe   | 是否试探关系状态或距离变化   |
| has_positive_reciprocity | 是否存在正向承接或有限延续   |
| has_rejection_signal     | 是否存在拒绝、否定、后退、拒聊 |
| has_push_pull_pattern    | 是否存在推进与后退并存的结构  |
| has_sustained_coldness   | 是否存在持续性低投入与冷处理  |
| signal_summary           | 对高级信号的客观总结      |

这一协议与 V2 实施方案中 S3 的设计保持一致，正式成为后续规则引擎的直接输入。

---

### 3.3 完成 S3 Prompt 初版设计

今日围绕以下原则完成了 S3 Prompt 的初版设计：

* 只提取高级关系信号，不输出高层结论
* 严格 JSON 输出
* 不新增字段
* signal_summary 只保留客观信号描述
* 禁止把单次礼貌互动误判为 intimacy
* 禁止把单次简短回复误判为 sustained_coldness

Prompt 初版已能稳定覆盖：

* 亲密靠近
* 关系试探
* 正向承接
* 回避拒绝
* 拉扯模式
* 持续冷淡

这一步标志着 Day 3 从“协议设计”进入“节点调试”。

## 四、调试过程与关键修复

### 4.1 首轮运行结果：大体可用，但边界不稳

首轮测试共跑了 10 条样本，整体表现已具备可用性，但暴露出 3 类关键边界问题：

| 问题类型             | 具体表现                                    |
| ---------------- | --------------------------------------- |
| 礼貌热情误判为 intimacy | 单次夸赞、礼貌回夸被误识别为亲密信号                      |
| 推进+后退被压成冷淡       | 一方推进关系、另一方后退时，模型直接输出 sustained_coldness |
| 有限承接识别不足         | 对“顺势回应、追问、轻度接话”识别过于保守                   |

这些问题如果不修，会直接污染 Day 4 的规则引擎判断。

---

### 4.2 第一轮补丁：压 intimacy，抬 push_pull，放宽 reciprocity

针对首轮问题，今日对 S3 Prompt 做了第一次结构性修补，核心补丁包括：

* 单次夸赞、礼貌回夸、普通友好互动，不足以单独构成 has_intimacy_signal = true
* 若一方持续推进关系话题、另一方回避或后退，应优先识别 has_push_pull_pattern = true
* 只要对方对关系相关表达进行了顺势回应、追问或有限延续，即可视为 has_positive_reciprocity = true

这一轮修补后，S3 对高级关系信号的边界理解开始明显改善。

---

### 4.3 第二轮补丁：加入优先级与特别边界规则

在第二轮调试中，又进一步发现模型在少数样本上仍存在保守偏差，因此继续增强 Prompt，新增了：

#### ① 字段优先级

明确：

```text
push_pull 的优先级高于 sustained_coldness
```

即：
只要更符合“推进 + 后退”的拉扯结构，就优先判定为 push_pull，而不是直接判定为持续冷淡。

#### ② 并存规则

明确允许：

```text
has_positive_reciprocity = true
has_rejection_signal = true
has_push_pull_pattern = true
```

同时成立，避免模型误以为这些字段必须互斥。

#### ③ 特别边界规则

针对最容易误判的三类样本，补入专门规则：

* 单次夸赞 + 回夸 → intimacy=false，positive_reciprocity=true
* 明确想念/想见 + 对方先接一点后又退 → intimacy=true，positive_reciprocity=true，rejection=true，push_pull=true
* 一方持续追问关系变化 + 另一方持续回避 → relationship_probe=true，rejection=true，push_pull=true，sustained_coldness=false

这一轮补丁后，S3 开始具备更稳定的边界控制能力。

## 八、阶段成果总结

今日完成的不是简单“补 prompt”，而是正式把高级关系特征拆成可观察、可复用、可供规则系统读取的结构化字段。

* 礼貌热情 ≠ intimacy
